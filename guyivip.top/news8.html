ShareSim：虚拟股票交易系统前后端架构拆解

Apr 2026

ShareSim 是部署在我本人博客 guyivip.top 下的一个“附加模块”：它既要像普通静态页面一样可以被博客导航直达， 又必须具备用户注册登录、持仓与交易流水等动态能力，因此整体采用“静态站点 + 轻量后端”的组合架构。

1. 技术栈一览

  • 后端：Flask + Gunicorn
  • 数据：SQLite（单机、低成本、免运维）
  • 行情：Tushare（实时行情 + 盘后/历史数据）
  • 前端：多页 HTML + 原生 JavaScript（无框架、低资源消耗）
  • 部署：Nginx 监听 80/443，对外提供 HTTPS；反向代理到本机 127.0.0.1:8080 的 Gunicorn

2. 为什么采用这种架构

我本人博客是以静态文件为主，已在线上稳定运行。ShareSim 作为附属功能，核心诉求是“最小侵入、易部署、易维护”：

  • 不改主站架构：博客仍由 Nginx 提供静态文件；只在 /share/ 子路径接入后端。
  • 后端端口不暴露公网：Gunicorn 只监听服务器本机 127.0.0.1:8080，外网只走 443。
  • SQLite 免运维：不引入 MySQL/Postgres，减少资源占用与运维复杂度，适合低并发的个人站点。
  • 多页 + 原生 JS：避免引入前端工程化（构建、打包、依赖、Node 环境），上传即用。

4. 后端：Flask + Gunicorn（为什么要工厂模式）

后端使用 create_app() 工厂函数返回 Flask 实例，Gunicorn 启动入口使用： app:create_app()。工厂模式的好处是：

  • 部署标准化：Gunicorn/WSGI 直接调用工厂，避免 import 时副作用。
  • 初始化集中：数据库初始化、路由注册等在创建时完成。
  • 可测试：本地可以用测试客户端直接构造 app 做接口验证。

5. 数据：SQLite 表结构（为什么够用）

ShareSim 的用户规模小、并发低，SQLite 的单文件数据库足够支撑。 关键表包括：users（用户与资金）、holdings（持仓）、transactions（交易流水）。

这种设计让你可以把整个模块打包迁移：share/ 文件夹上传后，数据库文件也一起带走，部署成本极低。

6. 行情：Tushare 的实时与历史数据组合

行情分两类：

  • 实时：用于查价/下单前确认价格（优先走实时接口）。
  • 历史：用于 K 线（需要 token，适合拉取日线数据）。

生产环境建议设置环境变量（避免重启丢失）： 

export TUSHARE_TOKEN=你的token
export TUSHARE_PROXY=http://你的代理:端口/
export STOCKSIM_SECRET_KEY=一段随机字符串

7. 前端：多页 + common.js（为什么要做路径自适应）

ShareSim 部署在 /share/ 子路径，而不是站点根目录。前端统一通过 common.js 做：

  • 子路径 Base 自动识别（保证从任何页面跳转都不会 404）
  • 统一封装 api() 方法（自动拼接前缀、处理错误）
  • 统一鉴权跳转（未登录自动回到登录页）

8. 部署过程（Nginx 反代的意义）

生产部署的核心是：Gunicorn 只在本机监听，Nginx 负责外网入口与 HTTPS。 典型配置片段如下： 

location ^~ /share/ {
    proxy_pass http://127.0.0.1:8080/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}

这样做的好处是：

  • 安全：后端端口不暴露公网，减少被扫/被打风险
  • 统一 HTTPS：证书只在 Nginx 配一次，后端不处理 TLS
  • 与主站共存：主站静态资源不受影响，只新增一个子路径

9. 这套架构的边界与改进方向

  • 并发能力：Gunicorn sync worker 对小规模够用；更大并发可以改为 gevent/uvicorn（需评估依赖与改造成本）。
  • 数据一致性：目前下单价格以“下单时刻”为准，后续可加入滑点、手续费、成交规则。
  • 风控：可加入交易时间限制、涨跌停校验、最大持仓比例等。
  • 观测性：可加入结构化日志、错误上报与监控（但会增加运维成本）。
